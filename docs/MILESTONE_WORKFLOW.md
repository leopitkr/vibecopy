# 마일스톤 완료 시 cursor.rule 준수 절차

**목적:** 단계별 진행 완료 시 `VibeCopy_Development_Milestones.md` 체크박스 표시 + 커밋 해시 기록을 **반드시** 지키기 위함.  
**근거:** `require/cursor.rule` §5, §6.2, §6.3.

---

## 1. 왜 안 지켜졌는지 분석

### cursor.rule에서 요구하는 것

| 절 | 요구 사항 |
|----|-----------|
| **§5.1** | 체크박스 완료 시: `- [ ]` → `- [x]` 로 변경하고, **같은 줄에** `(commit: <SHORT_HASH>)` 를 붙인다. |
| **§5.2** | 원칙: **체크박스 1개 = 커밋 1개**. 예외: 같은 소섹션에서만 최대 3개까지 번들 가능. 번들 시 **모든 체크된 줄에 동일한 커밋 해시** 기록. |
| **§6.2** | 커밋 후: **short hash를 복사해 마일스톤 해당 줄에 붙인다.** (자동화해도 이 단계는 명시적으로 수행) |
| **§6.3** | **필수:** 완료한 체크박스마다 `pnpm finish` 로 마무리. 이 스크립트가 (1) 기능 커밋, (2) 마일스톤에 해시 기록한 docs 커밋 **두 개**를 만든다. |

### 실제로 벌어진 일 (Generate UI 작업 예시)

- **§6.3 위반:** `pnpm finish` 를 **한 번도 실행하지 않음**.
- 수동으로: (1) 기능 커밋 한 번, (2) 마일스톤 파일에서 체크박스 6개 + 해시 직접 수정 후 docs 커밋.
- 결과적으로 **해시가 들어간 형태**는 맞지만, **“pnpm finish 로 마무리”** 라는 규정은 지키지 않음.

### 원인 요약

1. 작업 지시가 “마일스톤 수동 업데이트 + 커밋” 위주로 되어 있었고, **cursor.rule §6.3의 pnpm finish 필수**를 전제로 하지 않았음.
2. 번들(체크박스 여러 개를 한 커밋으로 처리) 시, **finish 는 한 번만 호출하고 나머지 줄은 같은 해시로 수동 보정**해야 한다는 절차가 정리돼 있지 않았음.

---

## 2. 앞으로 반드시 할 절차 (체크리스트)

단계별 진행이 **완료될 때마다** 아래를 따른다.

### A. 체크박스 1개만 완료한 경우

1. 구현 + `docs/features/...` 작성/수정 후 `pnpm lint` (필요 시 `pnpm format`) 실행.
2. **반드시** 아래를 실행 (체크박스 문구는 **마일스톤 파일에 있는 문자열과 완전히 동일**하게).

   ```bash
   pnpm finish --item "<정확한 체크박스 문구>" \
     --milestone "require/VibeCopy_Development_Milestones.md" \
     --doc "docs/features/<phase>/<slug>.md" \
     --msg "<기능 커밋 메시지>"
   ```

3. 실행 후 확인:
   - `git log -n 2` 에 커밋 2개 (기능 커밋 → docs 커밋).
   - `require/VibeCopy_Development_Milestones.md` 에 해당 줄이  
     `- [x] ... (commit: <실제_짧은해시>)` 형태인지 확인 (placeholder 없어야 함).

### B. 같은 소섹션에서 2~3개 번들한 경우 (§5.2 예외)

1. 구현 + `docs/features/...` 작성 후 `pnpm finish` 를 **번들 중 하나의 체크박스**에 대해서만 **한 번** 실행.
2. 생성된 **기능 커밋의 short hash**를 확인.
3. 마일스톤 파일을 열어, **나머지 번들 항목**의 줄도 같은 해시로 수정:
   - `- [ ] 다른 항목` → `- [x] 다른 항목  (commit: <동일_해시>)`
4. 수정한 마일스톤 파일만 스테이징 후 커밋:
   - 예: `docs: record commit hash for milestone - 나머지 항목들`

### C. 4개 이상 번들한 경우 (예: UI 6개 한 번에)

- **§5.2** 는 “최대 3개까지 번들”을 허용하므로, 4개 이상은 원칙적으로는 체크박스당 커밋을 나누는 것이 규정에 맞음.  
- 다만 이미 “한 번에 구현 후 6개 체크”로 진행한 경우, **해시라도 누락 없이** 넣는 것이 우선이므로:
  1. `pnpm finish` 를 **한 항목**에만 실행해, 기능 커밋 + 해당 줄 해시 기록 docs 커밋을 만든다.
  2. 마일스톤에서 **나머지 5개 줄**을 같은 기능 커밋 short hash로 수정하고, 한 번에 스테이징 후 docs 커밋 한 번 더 (예: `docs: record commit hash for milestone - UI 번들 나머지 항목`).

---

## 3. 실행 규칙 (cursor.rule §6.3 마지막 문단)

- **명령만 제안하지 말고, 실제로 실행한다.**
- 터미널에서 `pnpm finish` 및 필요 시 마일스톤 수동 보정 후 `git add` / `git commit` 까지 수행하고,  
  `git log -n 2` 또는 해당 줄 내용을 보여주어 **완료 증명**을 한다.

---

## 4. 요약

- **단계별 진행이 완료되면:**  
  `VibeCopy_Development_Milestones.md` 에서 해당 항목을 `- [x]` 로 바꾸고, **같은 줄에** `(commit: <SHORT_HASH>)` 를 붙인다.
- **해시 기록은:**  
  가능한 한 **항상 `pnpm finish` 를 한 번 이상 사용**하고, 번들 시에는 나머지 줄을 **동일 해시로 수동 보정**한 뒤 docs 커밋으로 남긴다.
- 이렇게 해야 cursor.rule §5, §6.2, §6.3 을 모두 지키게 된다.
